# 10 — Decisions

> Resolved decisions, open questions, decision log.

---

## Resolved Decisions

| # | Question | Decision | Rationale |
|:--|:---------|:---------|:----------|
| 1 | Service interfaces in core? | **Deferred.** Zero interfaces initially. Promote when shared by 2+ servers. | Promoting = minor bump. Demoting = breaking. Start conservative. |
| 2 | All storage providers in core? | **Yes.** | Runtime serverless whitelist gates loading. Splitting into separate packages adds coordination overhead with no real benefit since heavy deps are optional. |
| 3 | Config extension pattern? | **Separate schemas, not merged.** | Core validates infrastructure. Servers validate domain. No `z.merge()` or `defineConfig()` needed. See [03-config-container.md](03-config-container.md). |
| 4 | CLAUDE.md management? | **Core ships `CLAUDE.md` in package.** Server references it. | Agent reads from `node_modules/`. See [05-agent-dx.md](05-agent-dx.md). |
| 5 | Core's `package.json` identity? | **`createApp()` accepts `name`/`version` overrides.** | Config module derives `mcpServerName`/`mcpServerVersion` from `package.json` via import assertion ([config/index.ts:16](../src/config/index.ts#L16)). Precedence: `createApp({ name })` > `MCP_SERVER_NAME` env var > core's `package.json`. Config module's `parseConfig` gains `overrides` parameter. |
| 6 | Scheduled worker handlers? | **Callback pattern via `onScheduled`.** | Signature includes `ExecutionContext` for `ctx.waitUntil()`. See [02-public-api.md](02-public-api.md). |
| 7 | `@/` alias in server code? | **Two import styles, self-documenting.** | `@/` = server's `src/`. `@cyanheads/mcp-ts-core/*` = framework. Clear which code is "mine" vs. "framework." |
| 8 | Template repo identity after extraction? | **`mcp-ts-template` npm gets final major with deprecation.** | GitHub repo transforms in-place. New thin template repo created as consumer. See [01-architecture.md](01-architecture.md). |
| 9 | `setup` callback async support? | **`(core: CoreServices) => void \| Promise<void>`.** | `createApp()` awaits the result. Supports async init (DB connections, API warm-up, remote config). |
| 10 | `ServerHandle` surface area? | **Expose `services` (read-only).** *(Updated by #15 — no DI container.)* | Downstream servers need service access for integration testing, health checks, and programmatic embedding. `ServerHandle.services` is typed as `CoreServices` (read-only). |
| 11 | `pino-pretty` placement? | **`devDependencies` only.** | Currently in `dependencies` ([package.json:186](../package.json#L186)) — must be moved to `devDependencies` before extraction. Already dynamically resolved via `require.resolve()` with try/catch fallback to JSON output ([logger.ts:107-112](../src/utils/internal/logger.ts#L107-L112)). Servers that want pretty dev output install it themselves. |
| 12 | `hono` as peer dependency? | **Core dependency only, not peer.** | Core owns the Hono version. Servers extending the HTTP transport use the version core provides. If version conflicts arise in practice, promote to peer in a minor release. |
| 13 | Provider code in core without interfaces? | **Yes — intentional.** | Storage and service providers ship in core. Their heavy deps (`@supabase/supabase-js`, `openai`) are Tier 3 optional peers. The lazy import pattern means provider code is inert until activated by config. Service interfaces stay in downstream servers until shared by 2+ servers. |
| 14 | Zod peer dep version range? | **`"zod": "^4.3.0"`.** | `^4.0.0` is too broad — earlier Zod 4 releases had API churn. Pin minimum to `4.3.0`, the version the codebase is tested against. |
| 15 | DI container vs `createApp()`? | **Replace container with `createApp()` direct wiring.** | The container has 7 production `resolve()` calls (2 in `index.ts`, 4 in `server.ts`, 1 in `transports/manager.ts`). Zero tool/resource/prompt definitions use it — they receive context via function parameters. The dependency graph is static, linear, and small (~15 services). Direct construction in `createApp()` is explicit, debuggable with a stack trace, and eliminates the token/registration/resolve indirection. Server-specific services use module-level lazy accessors (already the established pattern). `setup()` callback replaces `services(container)`. |
| 16 | Internal vs external documentation/skills? | **Explicit audience tagging.** | Core-extraction docs are internal (building the core). Skills and CLAUDE.md shipped in the package are external (consumer-facing). Skill metadata gets an `audience` field (`internal` or `external`). Internal skills stay in the core repo; external skills ship in the package and get copied to consumer projects. |
| 17 | Unified `Context` vs split `appContext`/`sdkContext`? | **Unified `Context`.** | Tool authors interact with one object, not two. `Context` wraps `RequestContext` + SDK `RequestHandlerExtra` into a single facade with `log`, `state`, `elicit?`, `sample?`, `progress?`, `signal`. Internally, `RequestContext` and the global logger still exist. See [12-developer-api.md](12-developer-api.md). |
| 18 | Builder functions vs raw definition objects? | **Builder functions as primary API.** `tool()`, `resource()`, `prompt()` produce `ToolDefinition`/`ResourceDefinition`/`PromptDefinition`. | Shorter field names (`input`/`output`/`handler`/`format` vs `inputSchema`/`outputSchema`/`logic`/`responseFormatter`), name as first arg, inline auth. Builders are zero-runtime sugar — they just restructure arguments. Raw types remain for power users. |
| 19 | Inline auth vs `withToolAuth` HOF? | **Inline `auth` property.** | `auth: ['scope']` on the definition. Handler factory checks scopes before calling handler. `withToolAuth`/`withResourceAuth` removed. `checkScopes(ctx, scopes)` exported from `./auth` for dynamic scope computation. |
| 20 | `task: true` flag vs separate `TaskToolDefinition`? | **`task: true` flag.** | Developer writes a normal handler; framework manages task lifecycle (create, background run, progress, store result). ~80 lines of task boilerplate becomes `task: true`. Raw `TaskToolDefinition` with manual `taskHandlers` preserved in `./tasks` as escape hatch. |
| 21 | Flatten `createApp()` options? | **Yes.** `tools`, `resources`, `prompts` at top level. | `definitions: { tools, resources, prompts }` nesting removed. Less indentation, easier to read. All arrays default to `[]`. |
| 22 | `ctx.state` tenant defaulting in stdio? | **Default `tenantId` to `'default'` for stdio.** | Stdio is single-tenant. Requiring explicit tenantId for state operations in stdio mode is unnecessary friction. HTTP with auth derives tenantId from JWT `'tid'` claim as before. |
| 23 | Logger init before `setup()`? | **Yes.** Move logger initialization before `setup()` in the boot sequence. | `setup()` may need to log (e.g., "Initializing database connection..."). Current boot sequence inits logger at step 8, after `setup()` at step 4. Reorder so logger is usable in `setup()`. `setup()` uses `core.logger` (global); request handlers use `ctx.log` (scoped). |
| 24 | Linter for consumers: ESLint or Biome? | **Biome.** | Core uses Biome. Consumer examples should match. Replace `./eslint.config` export with `./biome.json` in subpath exports. Servers extend core's Biome config. |
| 25 | Handler field name: `handler` vs `logic`? | **`handler`.** | "Logic throws, handlers catch" was an internal framework distinction. Consumers write handlers. The framework's catching layer is invisible to them. `handler` aligns with every other web framework (Express, Hono, FastAPI). |
| 26 | `vitest.config` shipping format? | **Author as plain `.js`, not compiled from `.ts`.** | Vitest configs are simple objects that don't benefit from TypeScript. `.ts` files can't be imported from `node_modules` through subpath exports without extra consumer-side TypeScript loader config. `.js` works universally. Core's own `vitest.config.ts` (development use) is separate from the exported `vitest.config.js` base preset. |
| 27 | Phase 1 scope? | **Split into 1a (fixes & hardening) and 1b (DI removal).** | 1a is low risk — dep placement, coupling fixes, test cleanup. 1b is medium risk — central wiring change. Splitting isolates blast radius: easy wins land first, DI removal gets a clean start. If 1b surfaces surprises, test work from 1a is already committed. |
| 28 | `ctx.state.list()` pagination? | **Accept cursor + limit params.** | `list(prefix?, opts?: { cursor?, limit? })` returns `{ items, cursor? }`. Without cursor input, consumers can see "there's more data" but can't fetch page 2. The convenience facade must match `StorageService`'s pagination capability. |
| 29 | Worker build validation in examples CI? | **Yes — add Worker bundle step.** | Bundler resolution of subpath exports from `node_modules` is the most fragile integration path. If it breaks, downstream Worker deployments break. CI must run `bun build` with `--no-external` against the examples `worker.ts` to catch this early. |
| 30 | Phase 6 rename sequencing? | **Rename first, then create new repo.** | Can't create `cyanheads/mcp-ts-template` while the old repo still has that name. Sequence: (1) npm deprecation notice, (2) rename GitHub repo `mcp-ts-template` → `mcp-ts-core`, (3) create new `mcp-ts-template`. |

## Open Questions

| # | Question | Leaning | Notes |
|:--|:---------|:--------|:------|
| 1 | Exports catalog format: hand-maintained or auto-generated? | **Hand-maintained initially.** Graduate to generation once exports stabilize. | Generated (via build step extracting JSDoc + export names) can't drift. But premature until the export surface is stable. |
| 2 | `extraEnvBindings`/`extraObjectBindings` typing? | **Defer to 1.0.** | Both are `Array<[string, string]>` which loses type info. Generic `createWorkerHandler<B extends CoreBindings>` could enforce at compile time. Possibly overengineered for 0.1. |
| 4 | `@modelcontextprotocol/ext-apps` — keep as Tier 1 or demote? | **Gate on Phase 5 (0.1.0 publish).** | Still experimental as of 2026-03. If still pre-stable at Phase 5 publish, demote to Tier 3 optional peer before the first public release. Don't couple core's stability to an unstable dependency from day one. Added as pre-publish gate in [09-execution.md](09-execution.md) Phase 5. |
| 5 | `examples/` in the published npm package? | **Exclude.** Resolve now. | Useful as runnable reference but adds package size. The reference template repo is the user-facing example; `examples/` exists for core's CI. |
