# 0.6.6 — 2026-04-22

HTTP-mode servers deployed behind a TLS-terminating reverse proxy (Cloudflare Tunnel, Caddy, nginx, ALB) previously emitted `http://` self-referential URLs from the landing page, SEP-1649 Server Card, and RFC 9728 protected-resource metadata — because every handler derived `baseUrl` from the inbound request, which reflected the proxy-to-container hop rather than the public `https://` edge. Every copy-paste connect snippet was a dead link. This release adds an `MCP_PUBLIC_URL` override that is used verbatim as the base URL when set.

The `design-mcp-server` skill steps up to v2.6 with a codified 3-word `{server}_{verb}_{noun}` tool naming convention, a simplified shape taxonomy (Workflow and Instruction as explicit variants with distinct design pressures, single-action tools as the implicit default), documented workflow safety patterns (elicit-guarded destructive modes, safe defaults on blast-radius parameters), an "irreversible operations stay in the UI" principle distinct from `destructiveHint`, and a surface-audit pass. The `add-tool`, `field-test`, and `polish-docs-meta` skills update in lock-step — all tool-name examples follow the 3-word shape.

Closes [#43](https://github.com/cyanheads/mcp-ts-core/issues/43) and [#44](https://github.com/cyanheads/mcp-ts-core/issues/44).

## Added

- **`MCP_PUBLIC_URL` env var** — optional public-facing origin override (e.g. `https://mcp.example.com`). When set, the landing page connect snippets, SEP-1649 Server Card `endpoints.streamable_http`, and RFC 9728 `resource` fallback use this origin instead of the inbound request URL. Required behind TLS-terminating reverse proxies where the container sees `http://` but users reach the server over `https://`. Leave unset for direct (non-proxied) deployments; behavior is unchanged in that path. Surfaced on `AppConfig.mcpPublicUrl` (Zod-validated URL, empty-string → `undefined`) and on `ManifestTransport.publicUrl` (only present when configured). Wired through `CloudflareBindings` and `CORE_ENV_BINDINGS` in `createWorkerHandler()` so a plain `MCP_PUBLIC_URL` binding activates the override in Worker deployments without manual `extraEnvBindings` wiring. [#43](https://github.com/cyanheads/mcp-ts-core/issues/43)

## Changed

- **`skills/design-mcp-server/SKILL.md` → v2.7** — designed for the full range of MCP servers: simple APIs, complex multi-API wrappers, no-API utilities (computation, text/code tools), and stateful server-as-service designs. Tool naming relaxed from a strict 3-word rule: 3-segment `{server}_{verb}_{noun}` remains the strong default, 2 is valid when the operation name is canonical (`git_pull`, `git_status`), 4 is valid when the noun is inherently compound (`patentsview_search_patent_families`) — don't pad with a filler word. Shape taxonomy: Workflow and Instruction as explicit variants, single-action tools as the implicit default. Mode consolidation collapsed from a full subsection (code example + consolidate-vs-split table) into a one-line tip noting it affects both naming (noun-led, e.g. `github_pull_request`) and handler dispatch. Step 4's conditional subsections (Instruction tools, Workflow tool safety, Batch input, Convenience shortcuts) open with `**Applies when:**` triggers so smaller models don't treat every subsection as mandatory; always-apply subsections stay untagged. Workflow-tool safety covers elicit-guarded destructive modes (`mode === 'apply'`) with `destructiveHint` annotation fallback and safe defaults on blast-radius parameters (`mode: 'preview'`, `dryRun: true`, explicit `confirmCount`). "Irreversible operations stay in the UI" principle distinct from `destructiveHint`, with examples spanning database tables, versioned cloud-storage buckets, workspace admin-role revocation, GDPR permanent-delete, analytics-warehouse partition purges, and email audiences. Granularity broadened — tools can be a single API call, a multi-step workflow, or internal-only; don't constrain by call count. Step 1 tagged conditional (skip for internal-only servers) and research lead flipped — inline by default, parallel agents reserved for genuinely parallelizable multi-API research. Inputs broadened to "Data sources / source of truth" so in-memory state and pure computation are valid answers. Step 7 adds a "Server-as-service" paragraph pointing stateful designs at `ctx.state` for tenant-scoped KV (resilience and API-efficiency tables explicitly don't apply). Checklist split by applicability: unconditional items first, conditionals prefixed with `**If X:**`; "5–10 outcomes" softened to "3–10, scaled to domain size". Examples diversified across database migrations, deploy/release (canary/promote/rollback), observability incident triage, cloud infra, identity, and internal-only tools (`markdown_format_text`, `regex_test_pattern`, `tokens_count_text`) — previously clustered on email/notifications. Workflow Analysis's ASCII call-flow sketch replaced with a markdown table (`deploy_release` example); all ASCII diagrams removed. Auth scope examples aligned with the canonical `tool:<snake_tool_name>:<verb>` / `resource:<kebab-resource-name>:<verb>` patterns used across the framework. ## Fixed

- **Landing page, Server Card, and RFC 9728 protected-resource metadata now honor `MCP_PUBLIC_URL`** — `createLandingPageHandler`, `createServerCardHandler`, and `protectedResourceMetadataHandler` prefer the configured public URL over the inbound request origin. Trailing slash on `MCP_PUBLIC_URL` is stripped. Tests cover the proxied-deployment path (inbound `http://internal.container` + configured `https://mcp.example.com` → emitted URL carries the public origin) for all three surfaces, plus the trailing-slash normalization for the `resource` fallback. [#43](https://github.com/cyanheads/mcp-ts-core/issues/43)
- **Worker: `MCP_PUBLIC_URL` binding propagates to `process.env`** — the binding is now injected into `process.env` at Worker request time so the config layer's env-var read sees it correctly. Without this propagation, a bound `MCP_PUBLIC_URL` variable was invisible to the URL-override logic shipped above.
